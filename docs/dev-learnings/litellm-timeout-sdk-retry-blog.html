TL;DR — litellm에 timeout을 넘기면 OpenAI SDK의 httpx request timeout이 되고, timeout 시 SDK가 자동 retry(기본 2회)를 시도한다. LLM은 거의 다 만들었는데 매번 처음부터 다시 시작하느라 시간만 낭비하고 있었다. timeout 제거 + num_retries=0 명시 후 79초 만에 성공.
문제 상황

timeout을 아무리 늘려도
계속 실패한다

JIRAvis의 이메일 시각화 기능 — LLM에게 마크다운 브리핑을 보내면 HTML 이메일로 변환. 새벽 배치에서는 잘 되는데, "테스트 발송" 버튼을 누르면 HTTP 504 Gateway Timeout.

"nginx timeout 문제겠지." 동료 3명이 문서를 쓰고, 원인을 분석하고, timeout을 조정했다. 자신감에 차서 사내 테스트를 돌렸다.

Timeout 설정 vs 실제 대기 시간

진짜 원인

SDK의 이중 timeout + 자동 retry

timeout이 두 군데에 걸려 있었다. litellm에 전달하는 값과 asyncio.wait_for. litellm의 timeout은 OpenAI SDK → httpx request timeout으로 그대로 전달된다.

# openai/_constants.py
DEFAULT_TIMEOUT = httpx.Timeout(timeout=600, connect=5.0)
DEFAULT_MAX_RETRIES = 2 # retry 2회!

악순환의 메커니즘

timeout=100s 설정 시 실제로 일어나는 일:

0초
LLM 호출 시작
timeout=100s 설정. LLM: "긴 HTML 생성 중..."
100초
httpx가 연결을 끊는다
"100초 지났다! 연결 끊어!" → SDK TimeoutError 발생
100초
SDK가 자동 재시도
"에러네? max_retries=2니까 재시도!" → 처음부터 다시 시작
105초
asyncio.wait_for가 전체 취소
"100 + 5초 buffer 끝!" → asyncio.TimeoutError → fallback HTML
LLM은 매번 거의 다 만들었는데, 완성 직전에 SDK가 연결을 끊어버리고 처음부터 다시 시작시켰다.
LLM이 느린 게 아니었다. SDK가 몰래 시시포스를 시키고 있었다. 🪨
해결

timeout 제거 + retry 비활성화

BEFORE
kwargs = {
  "model": config.llm.model,
  "messages": [...],
  "timeout": timeout_s,
  # ↑ SDK httpx timeout
  # → retry 유발
}
AFTER
kwargs = {
  "model": config.llm.model,
  "messages": [...],
  "num_retries": 0,
  # timeout 미전달
  # → SDK 기본 600초
}
# asyncio.wait_for만으로 deadline 제어
response = await asyncio.wait_for(
    litellm.acompletion(**kwargs),
    timeout=timeout_s,  # 우리의 유일한 deadline
)
4차 테스트: 79초 만에 12,237자 HTML 생성 성공.
Retrying request 로그도 없다. LLM은 처음부터 충분히 빨랐다.
교훈

다음에 가져갈 것들

01
timeout은 한 레이어에서만 제어하라
여러 레이어가 각자 timeout을 가지면, 안쪽 timeout이 먼저 발동해서 바깥 의도와 다른 동작을 유발한다. timeout은 가장 바깥 레이어에서 한 번만 거는 것이 원칙.
02
SDK의 기본 retry 정책을 반드시 확인하라
OpenAI SDK의 DEFAULT_MAX_RETRIES = 2는 일반 API 호출에서는 합리적이지만, 수십 초~수분 걸리는 긴 생성 작업에서는 "거의 완성된 작업을 버리고 처음부터 다시"라는 재앙이 된다.
03
"timeout을 늘리면 해결되겠지"는 위험한 가정
"더 늘리면 되겠지"가 아니라 "왜 매번 정확히 timeout 시점에 실패하지?"라는 질문을 더 일찍 했어야 했다. 로그에 Retrying request가 찍히고 있었는데, 그 한 줄이 근본 원인의 직접적 증거였다.
04
동료 코드 리뷰의 위력
동료 CX의 "timeout 빼도 SDK retry는 남는다. num_retries도 명시적으로 꺼야 한다"라는 한 줄 피드백이 없었으면 num_retries=0을 빠뜨릴 뻔했다. 혼자 디버깅하면 80점, 동료 리뷰까지 하면 100점.
