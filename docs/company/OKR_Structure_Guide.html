2Prefix 공통 문법

[ 소속 · Objective-N · Key-N ] — 단 하나의 문법

모든 JIRA Summary 접두어는 단 하나의 공통 문법 하나로 통일됩니다. 이것이 Structure 의 설계 언어입니다. 왼쪽일수록 상위 레벨, 오른쪽일수록 세부 분류.

[ 소속  Objective-N  Key-N ]
1. 소속
AIE · AIE-{letter}
AIE = 파트. AIE-{letter} = 파트원 (letter ∈ {δ, ξ, λ, ψ, σ, π, θ, ζ, γ}) — 한 번 배정된 Greek letter 는 평생 고정.
2. Objective
O-N (N = 1..5)
GHRP 5개 평가 항목과 1:1 대응. 모든 Summary 에 필수로 포함됩니다 (Epic 제외).
3. Key (선택)
K-n (n = 1..4)
핵심제품 번호. O-1 (조직목표달성) 에만 부착되는 꼬리표 — 제품별 성과 집계를 위해.

5Bottom-Up · Level 3
팀 레벨

파트 Objective 이 팀 Epic 에 자연스럽게 귀결

Task 하나를 완료하면 → 개인 Story → 파트 Story → 팀 Story → 팀 Epic 까지 자동으로 progress 가 전파됩니다. 당신은 맨 아래 Task 하나만 잘 하면 됩니다.

[E4] Avatar 7000명 양성 ← 팀 Epic (이미 존재) └─ SWINNOTEAM-1274: Document Agent 개발 ← 팀 Story (MAU ≥ 5,000명) └─ [AIE O-1] 2026 조직목표달성 ← 우리 파트의 P-1 └─ [AIE O-1 K-n] K-1..K-4 제품 Story └─ [AIE-{letter} O-1 K-n] 9명의 파트원 M-1 Story └─ Task / Sub-task ← 당신의 일상 업무
Task 하나가 올라가면 → 개인 Story 진행률 ↑ → 파트 Story ↑ → 팀 Story ↑ → 팀 Epic 달성률 ↑.
당신의 Objective 만 신경쓰세요. rollup 은 Structure 가 자동 처리 합니다.

혁신/개선(P-2), 조직시너지(P-3), 역량강화(P-4) 는 팀 Epic 에 직접 연결되지는 않지만 E1, E2, E3 와 방향성 일치로 간접 기여 합니다. 50% 항목인 O-1 만 팀 Story 직속으로 두고 나머지는 파트 독립 영역으로 분리 — 평가 관점의 명확성을 위한 의도적 설계입니다.

6Structure 활용

JQL 검색 치트시트 — 질문 → 쿼리 → 결과

Prefix 규칙만 알면 별도 Label/Component 없이 Summary 접두어만으로 누구나 즉석 JQL 쿼리를 만들 수 있습니다. 아래는 자주 쓰는 7가지 패턴.

Q1
O-1(조직목표달성)에 몇 명의 파트원이 일하고 있지?
summary ~ "O-1" AND summary !~ "AIE O-1"
→ [AIE-δ O-1 K-1], [AIE-λ O-1 K-2], ... 9개 Story 조회. 제품별 인원 분포가 한눈에 보임.
Q2
AIE-λ 는 이번 주에 어떤 일들을 했지?
summary ~ "-λ" AND updated >= -7d
→ λ 의 4개 Story (M-1~M-4) 와 하위 Task/Sub-task 중 최근 7일 업데이트 전부. 1:1 미팅 자료가 자동 생성.
Q3
혁신/개선업무(O-2) 쪽 파트 전체 진행률은?
summary ~ "O-2"
→ [AIE O-2] (파트) + [AIE-δ O-2]..[AIE-γ O-2] 총 10개 Story 로 파트 전체 혁신 활동 조회.
Q4
K-1(Agent App Store) 담당자들이 지금 뭐하고 있지?
summary ~ "K-1"
→ K-1 파트 Story + K-1 담당 파트원 M-1 Story + 하위 Task. 제품팀 스탠드업 자료로 즉시 사용.
Q5
역량강화(O-4) 자격증/특허 계획을 모은 리스트 줘
summary ~ "O-4" AND issuetype = Story
→ 9명 + 파트 Story 총 10개. 각 Description 의 자격/특허/어학 목표가 SCI 리뷰 자료 로 export 가능.
Q6
지난 분기 AIE-δ 의 조직시너지(O-3) 기여 건수 집계
summary ~ "-δ" AND summary ~ "O-3" AND status = Done AND resolved >= -90d
→ δ 의 O-3 하위 Task 중 완료된 것만. 평가 시즌 정량 지표 확보 10초.
Q7
우리 파트의 수명업무(O-5) 비중이 얼마나 되지?
summary ~ "AIE O-5"
→ 본 Story 하위 Task 개수·평균 소요시간 집계 → 비계획 업무 비중 산출. OKR 재조정 근거 데이터.
핵심: 위 7가지는 전부 접두어 조합 만으로 만들어집니다. Label/Component 를 따로 관리할 필요 없음 — Summary 접두어가 바로 살아있는 필터 인덱스 역할.
7마무리

당신에게 부탁드리는 것

1

본인의 Story 4개만 관리

조직목표달성 1, 혁신/개선 1, 시너지 1, 역량강화 1. 이게 1년 동안 당신이 신경쓸 전부입니다.

2

Task / Sub-task 는 자유롭게

이름·개수·쪼개는 방식은 당신의 재량. 규칙은 단 하나 — Story 하위로 Link.

3

Task 완료 시마다 Status 업데이트

rollup 은 Structure 가 자동 처리 — 당신은 자신의 Task 상태만 정확히.

4

월 1회 1:1 리뷰

본인 Story 를 펼쳐서 TL 과 함께 점검. TL 이 자료 준비.

5

수명업무는 [AIE O-5] 하위에

비계획 지시 업무도 Task 로 등록. 나중에 OKR 재조정의 근거가 됩니다.

"나는 내 Objective 에만 집중하면,
파트 성과 → 팀 성과로 자동 rollup 된다."

Structure 를 이해 할 필요는 없습니다. 당신이 Story 4개를 매월 꾸준히 업데이트 하면, Structure 가 당신의 기여를 증명하는 수치를 TL·임원에게 전달합니다.
