# Claude Skills, 써보니까 이게 답이더라

> **한 줄 요약** — Skills는 "토큰도 아끼고, 자연어로 매번 반복 지시하는 귀찮음도 없애주는" Claude 생산성 도구입니다. 써보면 영수증(/context)이 달라집니다.

## TL;DR

- Anthropic 생태계는 **MCP → Agent → Skill** 순으로 진화 중. 토큰 효율 면에서 Skill이 압도적.
- 같은 일을 자연어로 매번 풀어쓰는 것보다, Skill로 박아두면 **더 세밀하고, 더 일관되고, 팀과 공유 가능**.
- 매일 쓰는 스킬 5선: `gh:issue` → `gh:commit` → `gh:pr` → `gh:pr-reply` → `visualize`.

> 💡 **도구 중립 알림** — 아래 예시는 제가 쓰는 **Claude Code** 기준이지만, "재사용 가능한 작업 레시피"라는 **Skill 개념은 AI CLI 공통**입니다. **OpenCode 는 Agent/Command**, **Roo Code 는 Modes** 로 동일한 역할을 합니다. 회사에서 Claude를 못 쓰더라도 **패턴은 그대로 이식** 가능 — 구체 매핑은 [섹션 5](#5-써보고-싶으시면--한-줄-설치) 하단 표를 참고하세요.

---

## 1. 왜 Skill인가? — 한 마디로 "토큰"

처음 Skill을 써본 계기는 단순합니다. **토큰을 아끼고 싶어서**.

Anthropic의 플랫폼 진화를 따라가보면 의도가 명확합니다.

| 단계                             | 역할                  | Context 비용                                     |
| -------------------------------- | --------------------- | ------------------------------------------------ |
| **MCP** (Model Context Protocol) | 외부 도구/데이터 연결 | 스키마가 통째로 로드 — 도구가 많을수록 무거움    |
| **Agent** (Sub-agents)           | 전문 역할 분리        | 에이전트 목록·설명이 **상주** (있기만 해도 소모) |
| **Skill**                        | 작업 레시피           | 이름+설명만 상주, 본문은 **호출 순간에만** 로드  |

### 내 현재 세션 `/context` 실측

```
⛁ System prompt:   6.4k  (3.2%)
⛁ System tools:    7.0k  (3.5%)
⛁ Custom agents:   9.5k  (4.7%)   ← 에이전트는 "있기만 해도" 차지
⛁ Memory files:    1.3k  (0.7%)
⛁ Skills:         12.8k  (6.4%)   ← 메타만 상주, 본문은 온디맨드
⛁ Messages:        1.8k  (0.9%)
⛶ Free space:    128.2k (64.1%)
```

**포인트** — Skills 항목의 12.8k는 내가 설치한 **수백 개 스킬의 이름+설명 전체**입니다. 본문(수백 줄짜리 SKILL.md)은 `/gh:pr` 같이 실제로 호출할 때만 로드됩니다. "상주형 Agent" vs "온디맨드형 Skill" 의 구조적 차이가 곧 토큰 차이로 돌아옵니다.

> 직접 같은 작업을 자연어 → Agent → Skill 로 바꿔가며 돌려봤는데, **Skill 경로가 가장 적게 먹습니다.** 눈으로 확인 완료.

## 2. 내가 매일 쓰는 스킬 — "GitHub 파이프라인 4단"

자연어로도 다 됩니다. 그런데 Skill로 박아두면 얻는 게 3가지입니다.

1. **반복 입력 제로** — `/gh:pr` 네 글자로 끝
2. **품질 일관성** — 매번 같은 체크리스트로 실행
3. **팀 공유** — git에 체크인하면 동료도 즉시 사용

아래 4개는 이슈 등록 → 커밋 → PR → 리뷰 응대까지 **한 사이클을 자동화**합니다.

---

### 2.1 `/gh:issue` — 대화를 GitHub Issue로

```bash
# Before (자연어)
지금 너와 대화한 내용을 요약해서 현재 repository의 issue로 등록해. label은 enhancement 로 설정.

# After
/gh:issue
```

**핵심 3줄**

- 현재 대화를 **feature / bug / misc** 로 자동 분류해 구조화된 issue 본문 생성
- `gh issue create` 로 바로 등록 — 확인 프롬프트 없음
- issue는 PR 초안·블로그에서 재활용되므로 **과도하게 압축하지 않음** (근거·결정 보존)

---

### 2.2 `/gh:commit` — 이슈 자동 링크 커밋

```bash
# Before
지금 너와 작업한 내용을 커밋으로 만들어.

# After
/gh:commit
```

**핵심 3줄**

- `git status`/`diff`/`log` 를 병렬로 읽어 **repo의 기존 커밋 스타일을 그대로 따라감**
- 최근 대화나 인자에서 `#N` 을 찾아 `Refs #N` / `Closes #N` 자동 footer 추가
- **항상 새 커밋** — amend 금지, hook 스킵 금지

---

### 2.3 `/gh:pr` — 커밋 묶어 PR 생성

```bash
# Before
지금까지 작업한 커밋을 묶어서 PR 만들어. PR은 신규 브랜치에서 생성해.

# After
/gh:pr
```

**핵심 3줄**

- base..HEAD 범위의 **모든 커밋**을 반영한 PR 본문 작성 (HEAD 하나만 보지 않음)
- 필요 시 자동 push, 관련 이슈가 있으면 자동 링크
- 결과는 PR URL 한 줄 — 바로 브라우저로

---

### 2.4 `/gh:pr-reply` — 리뷰 코멘트 응대의 "예의"

```bash
# Before
동료가 PR#132에 리뷰를 달았어. 검토하고 타당하면 수용해.
리뷰 에티켓에 맞춰서 답글도 꼭 달아. 너는 답글 다는 것을 매번 놓치더라.

# After
/gh:pr-reply
```

**핵심 3줄**

- 모든 리뷰 코멘트(**봇 포함** — gemini/sourcery/copilot)를 수집해 하나씩 판단
- **수용** 시 코드 수정까지, **거절** 시 이유를 명확히 답글 — "조용한 무시"는 없음
- 현재 브랜치 PR 자동 감지, `/gh:pr-reply 123` 처럼 번호 지정도 가능

> 이 스킬은 내가 반복했던 **"답글 다는 걸 매번 놓치더라"** 는 잔소리를 규칙으로 박아버린 사례입니다. 자연어 지시의 휘발성을 Skill이 잡아주는 대표 예.

---

## 3. 요즘 최애 — `/visualize`

```bash
/visualize some-doc.md
```

**핵심 3줄**

- 어떤 내용이든 **단일 파일 HTML 시각화** 생성 — 슬라이드/인포그래픽/대시보드/플로우차트/타임라인…
- 라이트·다크 테마, 유틸 메뉴(PNG 다운로드·인쇄), 시맨틱 HTML 등 **평가 기준을 기본 충족**
- 생성 즉시 `wslview`/`xdg-open`/`open` 으로 **자동 실행**, `file://` URL까지 반환

**왜 최애냐면** — README, 회의 노트, 제품 아이디어가 들어오면 *5분 안에 공유 가능한 비주얼*이 나옵니다. 동료한테 "이거 봐봐" 링크 하나 던지면 설명 끝. 말보다 이미지가 빠르니까요.

예시는 직접 보세요: `claude/skills/visualize/examples/` (carousel, cheatsheet, poster, dashboard, pitch-deck 등 18종)

## 4. 결론 — "반복을 자산으로"

| 기준         | 자연어           | Agent                    | **Skill**                      |
| ------------ | ---------------- | ------------------------ | ------------------------------ |
| Context 소모 | 매 지시마다 누적 | 상주                     | **메타만 상주, 본문 온디맨드** |
| 재현성       | 그때그때 다름    | 에이전트 프롬프트에 의존 | **SKILL.md = 단일 소스**       |
| 팀 공유      | 위키·복붙        | 설정 파일 필요           | **git에 체크인하면 끝**        |
| 수정 난이도  | 기억과 싸움      | 에이전트 튜닝            | **md 파일 편집**               |

Skill의 본질은 **"AI에게 내 업무 규칙을 영속화하는 것"** 입니다.
오늘 자연어로 세 번 이상 반복한 지시가 있다면 — 그건 이미 Skill 후보입니다.

다음에 만나면 `/skill:creator` 한 번 해보시죠. 토큰 영수증이 다르게 나옵니다.

설치는 Claude Code 세션 안에서 두 줄이면 끝납니다.

```bash
/plugin marketplace add dEitY719/claude-skills
/plugin install visualize
```

- 첫 줄 — 제가 운영하는 **마켓플레이스**를 등록 (한 번만)
- 둘째 줄 — 원하는 스킬 설치 (`visualize`, `gh:issue`, `gh:pr` 등 개별 선택)

업데이트/제거도 같은 패턴:

```bash
/plugin update visualize
/plugin uninstall visualize
```

전체 카탈로그와 추가되는 스킬 목록은 여기서 확인:
👉 [`dEitY719/claude-skills` README](https://github.com/dEitY719/claude-skills)

**All-or-nothing 아닙니다.** 필요한 것만 골라 쓰세요. 써보다 별로면 한 줄로 지우면 됩니다 — 심리적 허들 제로.

### 📌 Claude를 못 쓰는 환경이라면 — 도구별 대응 치트시트

회사에서 **OpenCode / Roo Code** 만 허용되는 경우, 이 글의 "Skill 이라는 개념"은 그대로 가져가고 **이름표만 바꾸면** 됩니다.

| 이 글의 용어 (Claude Code) | OpenCode                               | Roo Code                                     |
| -------------------------- | -------------------------------------- | -------------------------------------------- |
| **Skill** (`SKILL.md`)     | **Agent** / **Command** (`.opencode/`) | **Mode** (`.roomodes` 또는 Global Modes)     |
| `/gh:pr` 같은 slash 호출   | `/agent <name>` · `/<command-name>`    | Mode 전환 후 해당 목적에 맞게 프롬프트       |
| 마켓플레이스 (`/plugin`)   | 팀 레포에 `.opencode/` 체크인          | `.roomodes` 파일을 git 공유 / Custom Modes   |
| Context 소모 특성          | Agent 메타 상주, 본문은 호출 시 로드   | Mode 정의는 경량, 선택 시 system prompt 교체 |

핵심 메시지는 동일합니다 — **"자연어로 세 번 이상 반복한 지시는 파일로 영속화하라."** 파일 위치와 호출 문법만 해당 도구의 규칙으로 맞추면 됩니다.
