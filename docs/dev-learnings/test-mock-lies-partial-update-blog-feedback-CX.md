# test-mock-lies-partial-update-blog 피드백

## 결론

동료 개발자 대상의 범용성과 흥미 유발 가능성을 기준으로 보면,
**`test-mock-lies-partial-update-blog-v2-general.md`가 더 적합합니다.**

이유는 명확합니다.

- 특정 도메인(Jira, 특정 프로젝트 명칭) 의존도가 낮아 더 많은 팀과 서비스 맥락에 바로 이식됩니다.
- 메시지가 "라이브러리 추상화의 타입 불일치", "Mock-실세계 불일치", "검증 선행"이라는 보편 패턴으로 정리되어 재사용성이 높습니다.
- 구조가 단순해 핵심 교훈을 빠르게 전달합니다.

## 문서별 평가

## 1) `test-mock-lies-partial-update-blog.md` (v1)

강점

- 실제 사건 서사가 살아 있어 몰입감이 높습니다.
- 실패-수정-재실패의 반복이 드라마를 만들어 읽는 재미가 있습니다.
- 구체 로그/상황이 있어 신뢰감이 있습니다.

약점

- Jira/atlassian-python-api 맥락이 길어, 해당 스택 경험이 없는 독자에게 진입 장벽이 있습니다.
- 감정 표현과 장식 요소가 많아 기술 핵심이 중간중간 흐려집니다.
- "내 사례" 비중이 커서 일반화된 실행 지침을 찾으려는 독자에게는 길게 느껴질 수 있습니다.

## 2) `test-mock-lies-partial-update-blog-v2-general.md` (v2)

강점

- 문제를 도메인 중립적으로 표현해 대부분의 백엔드/CLI/API 개발자가 즉시 연결할 수 있습니다.
- 코드 예시가 개념 증명에 집중되어 핵심 전달 속도가 빠릅니다.
- 결론이 재현 가능한 원칙으로 수렴되어 팀 내 공유 문서/스터디 자료로 쓰기 좋습니다.

약점

- v1 대비 서사적 긴장감이 줄어 "재미" 측면에서는 다소 평이할 수 있습니다.
- 실제 운영 환경의 압박감(사용자 영향, 디버깅 비용)이 덜 드러나 체감 임팩트가 약해질 수 있습니다.

## 최종 추천

배포/공유용 메인 글은 **v2를 채택**하고, 아래 두 요소를 보강하면 가장 좋습니다.

- 도입부에 실제 실패 장면 3-4줄만 추가 (에러 메시지와 "왜 고쳤는데 안 됐는지"의 당혹감)
- 결론 직전에 "팀에서 바로 적용 가능한 체크리스트" 4-5개 추가

## 바로 적용 가능한 개선안 (v2 기준)

1. 도입부 훅 강화

- 첫 6-8줄 안에 "테스트 통과했는데 프로덕션 실패" 상황을 수치로 제시
- 예: "4회 재시도, 2개 필드 중 1개만 반영되는 부분 성공 버그"

2. 일반화 패턴을 더 선명하게

- "증상 변화 != 원인 해결"을 소제목으로 분리
- "Mock fidelity" 용어를 한 번 정의하고 반복 사용

3. 행동 지침형 마무리

- PR 체크리스트 형태로 종료
- 예: "실제 반환 payload 캡처 확인", "검증 후 실행 순서", "부분 업데이트 위험 시 롤백 전략"

4. 분량 최적화

- 현재 길이는 유지하되, 코드 블록은 핵심 블록만 남기고 설명 문장 1줄씩 보강
- 독자가 복붙 가능한 최소 예시를 우선 배치

## 한 줄 요약

`v1`은 몰입감이 강한 사례 글, `v2`는 재사용성이 높은 팀 학습 글입니다.  
동료 개발자 다수를 대상으로 한 "범용 + 흥미" 목표에는 **v2 우세**이며, v1의 현장감만 일부 이식하면 가장 완성도가 높습니다.
