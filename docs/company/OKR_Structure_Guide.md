## 1. 출발점 — GHRP 평가 항목을 OKR 언어로 번역

GHRP 에서는 다음 5개 항목으로 평가합니다.

| GHRP 평가 항목 | 비중 | 원래 의미 |
|----------------|------|-----------|
| 조직목표달성 | 50% | 핵심과제 달성 |
| 혁신/개선업무 | 20% | 프로세스 개선, Tool 제작 |
| 조직시너지 창출 | 20% | 타 조직 협업 기여 |
| 부서원/개인역량강화 | 10% | 교육·자격·특허·논문 |
| 수명업무 | tracking | 비계획 지시 업무 |

**설계 결정**: 각 평가 항목을 **하나의 Story** 로 만들어 **Objective** 로 취급합니다. 그 결과, 모든 JIRA Summary 접두어가 **단 하나의 공통 문법** 하나로 귀결됩니다.

### 1-1. Prefix 공통 문법 (확정)

모든 Summary 는 아래 한 문법으로 통일됩니다 — 이게 본 Structure 의 **설계 언어** 입니다.

```text
[ 소속   Objective-N   Key-N ]
   ↓         ↓            ↓
AIE      O-N (필수)     K-n (선택, O-1 에만)
AIE-{letter}   N=1..5   n=1..4
(파트 or 파트원)
```

**구성 요소**:

| 요소 | 값 | 의미 |
|------|-----|------|
| **소속** | `AIE` | 파트 자신 |
|       | `AIE-{letter}` | 파트원 (letter ∈ {δ, ξ, λ, ψ, σ, π, θ, ζ, γ}) |
| **Objective** | `O-N` (N=1..5) | GHRP 5개 평가 항목과 1:1 대응 (필수) |
| **Key** | `K-n` (n=1..4) | 핵심제품 번호 — **O-1 (조직목표달성) 에만** 부착 |

**문법 규칙**:
1. **왼쪽일수록 상위 레벨**, 오른쪽일수록 세부 분류 (읽기 순서 = 계층 순서)
2. **Key (`K-n`) 는 항상 브라켓 맨 끝** — 담당 제품은 "어느 Objective 의 세부 분기인가" 를 표시하는 꼬리표
3. **Epic 만 예외**: 파트 Epic 은 `[AIE M&V]` (Mission & Vision) — 최상위 미션이라 Objective 번호가 없음

### 1-2. 공통 문법의 5가지 실제 인스턴스

| # | 인스턴스 | 형식 | 예시 | 수량 |
|---|---------|------|------|------|
| 1 | 파트 Epic | `[AIE M&V]` | `[AIE M&V] 임직원이 체감하는 AI 효용성` | 1 |
| 2 | 파트 Story | `[AIE O-N]` | `[AIE O-2] 2026 혁신/개선업무` | 5 (N=1..5) |
| 3 | 파트 제품 Story | `[AIE O-1 K-n]` | `[AIE O-1 K-1] SLSI Agent App Store` | 4 (n=1..4) |
| 4 | 파트원 O-1 Story | `[AIE-{letter} O-1 K-n]` | `[AIE-δ O-1 K-1] 2026 조직목표달성` | 9 (인당 1개, 담당 제품 n) |
| 5 | 파트원 O-2~O-4 Story | `[AIE-{letter} O-N]` (N=2..4) | `[AIE-δ O-2] 2026 혁신/개선업무` | 27 (9명 × 3개) |

합계 **46개** — Epic 1 + 파트 Story 5 + 제품 Story 4 + 파트원 Story 36 (O-1: 9 + O-2~O-4: 27). 수명업무(O-5) 는 **파트 Story 한 개만** 존재하고 파트원 개별 Story 는 없음 — Task 만 동적 생성됩니다.

> **근거 문서**: PR [#147](https://github.com/dEitY719/dotfiles/pull/147) — `[AIE 파트]` 공통 prefix 의 가독성 문제를 해결하기 위해 OKR 업계 표준(`O` = Objective) 을 도입, 동시에 향후 KR 레벨(`[AIE O-1 KR-1.1]`) 확장 여지까지 확보.

이렇게 하면 **"평가받는 항목 = 일하는 단위 = Summary 접두어"** 가 1:1 매핑되어 별도 Label/Component 없이 접두어만으로 JQL 필터링이 가능해집니다 (→ §6 참조).

## 2. Level 1 — 당신의 Objective (개인)

**파트원 한 명은 Story 를 4개 가집니다**. 이게 당신이 1년 동안 신경 써야 할 전부입니다.

```text
파트원 δ (예시)
  ├─ [AIE-δ O-1 K-1] 2026 조직목표달성  (50%)  ← 담당 제품 K-1 의 핵심과제
  ├─ [AIE-δ O-2] 2026 혁신/개선업무 (20%)
  ├─ [AIE-δ O-3] 2026 조직시너지 창출 (20%)
  └─ [AIE-δ O-4] 2026 개인역량강화 (10%)
```

### 이 Story 안에서 당신이 자유롭게 하는 일

Story 는 **큰 틀** 입니다. 그 아래에 실제 업무를 **Task / Sub-task** 로 자유롭게 만드시면 됩니다.

```text
[AIE-δ O-1 K-1] 2026 조직목표달성  ← Story (당신의 Objective)
  ├─ Task: [4월] OKR 수립·승인          ← 월간 단위 task
  ├─ Task: [5월] 1차 KPI 점검
  ├─ Task: Feature A 개발                ← 구체적 업무
  │    ├─ Sub-task: API 설계
  │    ├─ Sub-task: 구현
  │    └─ Sub-task: 테스트
  └─ Task: Customer 인터뷰
```

**중요**: Task/Sub-task 는 **본인이 자유롭게** 만듭니다. 규칙은 하나 — **해당 Story 하위에 Link 되어야 합니다**. 그래야 평가 시점에 "이 일이 어느 Objective 의 성과인지" 자동으로 집계됩니다.

### Summary 의 letter 가 왜 Greek 문자인가?

δ, ξ, λ 같은 Greek letter 는 **실명 비노출** 을 위한 토큰입니다. Evaluation grade 같은 민감 정보를 JIRA Summary 에 실수로 노출하지 않기 위한 장치 — 실명 매핑은 TL 전용 `AIE_member_code_mapping.md` 에만 존재합니다.

**letter 는 평생 토큰** 입니다. 한 번 배정된 letter 는 변경하지 않습니다. 과거 JIRA 이력·평가 기록·검색 쿼리 전체가 letter 를 키로 연결되기 때문입니다.

## 3. Level 2 — 파트의 Objective (9명이 모여서)

당신의 Story 는 자동으로 파트 Story 의 하위로 연결됩니다.

```text
[AIE O-2] 2026 혁신/개선업무  ← 파트 Story (P-2)
  ├─ [AIE-δ O-2]  (당신)
  ├─ [AIE-ξ O-2]  (파트원 2)
  ├─ [AIE-λ O-2]  (파트원 3)
  ├─ [AIE-ψ O-2]  (파트원 4)
  ├─ [AIE-σ O-2]  (파트원 5)
  ├─ [AIE-π O-2]  (파트원 6)
  ├─ [AIE-θ O-2]  (파트원 7)
  ├─ [AIE-ζ O-2]  (파트원 8)
  └─ [AIE-γ O-2]  (파트원 9)
```

파트 Story (`[AIE O-N]`, N=1..5) 는 **9명의 개인 Story 를 포괄하는 컨테이너** 입니다. TL 은 파트 Story 레벨에서 OKR 을 수립하고 임원 승인을 받으며, 파트원은 그 하위 Initiative 에 자신의 개인 OKR 을 기여시키는 구조입니다.

### 특수 케이스 — 조직목표달성 (O-1) 은 제품별로 분기

조직목표달성은 파트원 Story 접두어 끝에 **담당 제품 번호 `K-n` 을 함께 표기** 합니다 (다른 O-N 은 K 없음).

```text
[AIE O-1] 2026 조직목표달성  ← 파트 Story (P-1)
  ├─ [AIE O-1 K-1] SLSI Agent App Store
  │   ├─ [AIE-δ O-1 K-1]  ← δ 가 K-1 담당
  │   └─ [AIE-ξ O-1 K-1]  ← ξ 가 K-1 담당
  ├─ [AIE O-1 K-2] SLSI Alpha Agent
  │   └─ [AIE-λ O-1 K-2]  ← λ 가 K-2 담당
  ├─ [AIE O-1 K-3] SLSI Cowork
  │   └─ [AIE-ψ O-1 K-3]  ← ψ 가 K-3 담당
  └─ [AIE O-1 K-4] MCP/Skill Hub
      └─ [AIE-σ O-1 K-4]  ← σ 가 K-4 담당
```

**왜 이렇게?** 조직목표달성은 제품 라인업 (K-1~K-4) 단위로 성과를 집계해야 하기 때문입니다. 제품별로 MAU, 활용률 등 KPI 가 다르므로 중간에 **제품 Story (K-n)** 를 끼워서 계층을 하나 더 둡니다.

## 4. Level 3 — 팀의 Objective (파트들이 모여서)

파트 Story 는 **팀 Epic** 및 **팀 Story** 에 연결됩니다.

```text
[E4] Avatar 7000명 양성  ← 팀 Epic (이미 존재)
  └─ SWINNOTEAM-1274: Document Agent 개발  ← 팀 Story (이미 존재, MAU ≥ 5,000명)
      └─ [AIE O-1] 2026 조직목표달성  ← 우리 파트의 P-1
          └─ K-1..K-4 제품 Story
              └─ 9명의 파트원 O-1 Story
```

혁신/개선 (P-2), 조직시너지 (P-3), 역량강화 (P-4) 는 팀 Epic 에 **직접 연결되지는 않지만 간접 기여** 합니다 (E1, E2, E3 와 방향성 일치). 이는 GHRP 평가 로직과 정렬하기 위한 의도적 분리 — 50% 항목인 O-1 만 팀 Story 직속, 나머지는 파트 독립으로 두어 **평가 관점의 명확성** 을 지킵니다.

**핵심**: 당신의 Task 하나를 잘 완료하면 →
- 당신의 Story (O-) 진행률이 올라가고 →
- 파트 Story (P-) 진행률이 올라가고 →
- 팀 Story (SWINNOTEAM-1274) 진행률이 올라가고 →
- 팀 Epic (E4) 달성률이 올라갑니다.

**당신은 당신의 Objective 만 신경쓰면 됩니다. rollup 은 Structure 가 자동 처리합니다.**

## 5. Prefix 규칙 정리 (치트시트)

공통 문법: **`[소속 Objective-N Key-N]`** — 왼쪽부터 계층 순(소속 → Objective → Key).

| 접두어 | 레벨 | 의미 | 예시 |
|--------|------|------|------|
| `[AIE M&V]` | 파트 Epic | 파트 미션·비전 (North Star) | `[AIE M&V] 임직원이 체감하는 AI 효용성` |
| `[AIE O-N]` | 파트 Story | 파트 Objective (N=1..5) | `[AIE O-2] 2026 혁신/개선업무` |
| `[AIE O-1 K-n]` | 파트 Story | K-1..K-4 제품 (P-1 하위, n=1..4) | `[AIE O-1 K-1] SLSI Agent App Store` |
| `[AIE-{letter} O-1 K-n]` | 파트원 Story | 개인의 조직목표달성 (제품 n 담당) | `[AIE-δ O-1 K-1] 2026 조직목표달성` |
| `[AIE-{letter} O-N]` | 파트원 Story | 개인의 Objective N (N=2..4) | `[AIE-δ O-2] 2026 혁신/개선업무` |

### 규칙 요약

1. **`O-N`** = Objective N — GHRP 5개 평가 항목과 1:1 대응, 항상 **소속 뒤** 에 옴
2. **`{letter}`** = 개인 식별자 (δ, ξ, λ, ψ, σ, π, θ, ζ, γ) — 평생 고정, 재사용 없음
3. **`K-n`** = 핵심제품 번호 (1..4) — O-1 (조직목표달성) 에만 붙음, 항상 **브라켓 맨 끝** 에 위치
4. **`[AIE ...]`** = 파트 레벨, **`[AIE-{letter} ...]`** = 개인 레벨 (하이픈이 개인 표시)

## 6. 이 Structure 로 얻는 이점 — 실제 JQL 검색 예시

> 모든 검색은 JIRA 의 Summary 필드 기반. 접두어만 알면 누구나 즉석 쿼리 가능.

### 6-1. "O-1 (조직목표달성) 에 몇 명이 일하고 있지?"

```jql
summary ~ "O-1" AND summary !~ "AIE O-1"
```
→ `[AIE-δ O-1 K-1]`, `[AIE-λ O-1 K-2]`, ... 9개 Story 가 나옴. **제품별로 몇 명이 붙어 있는지 한눈에** 확인 가능.

### 6-2. "AIE-λ 는 이번 주에 어떤 일을 했지?"

```jql
summary ~ "-λ" AND updated >= -7d
```
→ λ 의 4개 Story (O-1~O-4) 와 그 하위 Task/Sub-task 중 최근 7일 업데이트된 것 전부. **1:1 미팅 자료가 자동 생성** 됨.

### 6-3. "혁신/개선업무 (O-2) 쪽 파트 전체 진행률은?"

```jql
summary ~ "O-2"
```
→ `[AIE O-2]` (파트) + `[AIE-δ O-2]`..`[AIE-γ O-2]` (9명) 총 10개 Story. **파트 전체의 혁신 활동** 이 묶여서 조회됨.

### 6-4. "K-1 (Agent App Store) 담당자들이 지금 뭐하고 있지?"

```jql
summary ~ "K-1"
```
→ K-1 파트 Story `[AIE O-1 K-1]` + K-1 담당 파트원 O-1 Story `[AIE-{letter} O-1 K-1]` 들 + 그 하위 Task. **제품팀 스탠드업** 에 그대로 활용 가능.

### 6-5. "역량강화 (O-4) 자격증/특허 계획을 모은 리스트 줘"

```jql
summary ~ "O-4" AND issuetype = Story
```
→ 9명 + 파트 Story 10개. 각 Description 에 개인별 자격/특허/어학 목표가 있으므로 **SCI 리뷰 자료** 로 즉시 export 가능.

### 6-6. "지난 분기 AIE-δ 의 조직시너지 (O-3) 기여 건수 집계"

```jql
summary ~ "-δ" AND summary ~ "O-3" AND status = Done AND resolved >= -90d
```
→ δ 의 O-3 하위 Task 중 완료된 것만. **평가 시즌 정량 지표** 확보 10초.

### 6-7. "우리 파트의 수명업무 (O-5) 비중이 얼마나 되지?"

```jql
summary ~ "AIE O-5"
```
→ 본 Story 하위의 Task 개수·평균 소요시간 집계 → **비계획 업무 비중** 이 나옴. OKR 재조정의 근거 데이터.

### 핵심
