# 완료 전 검증

## 개요

검증 없이 작업 완료를 주장하는 것은 효율이 아니라 거짓이다.

**핵심 원칙:** 주장 전에 증거, 항상.

**이 규칙의 문구를 위반하는 것은 이 규칙의 정신을 위반하는 것이다.**

## 철의 법칙

```
최신 검증 증거 없이 완료를 주장해서는 안 된다
```

이 메시지에서 검증 명령을 실행하지 않았다면, 통과했다고 주장할 수 없다.

## 게이트 함수

```
어떤 상태를 주장하거나 만족을 표현하기 전에:

1. 식별: 이 주장을 증명하는 명령은 무엇인가?
2. 실행: 전체 명령을 실행한다 (새롭게, 완전하게)
3. 확인: 전체 출력을 읽고, 종료 코드를 확인하고, 실패 수를 센다
4. 검증: 출력이 주장을 뒷받침하는가?
   - 아니오: 증거와 함께 실제 상태를 보고한다
   - 예: 증거와 함께 주장을 보고한다
5. 그때서야: 주장을 한다

어떤 단계라도 건너뛰면 = 검증이 아니라 거짓이다
```

## 흔한 실패 사례

| 주장 | 필요한 것 | 충분하지 않은 것 |
|------|----------|----------------|
| 테스트 통과 | 테스트 명령 출력: 0 실패 | 이전 실행, "통과할 것이다" |
| 린터 클린 | 린터 출력: 0 오류 | 부분 검사, 추정 |
| 빌드 성공 | 빌드 명령: exit 0 | 린터 통과, 로그가 좋아 보임 |
| 버그 수정됨 | 원래 증상 테스트: 통과 | 코드 변경됨, 수정되었다고 가정 |
| 회귀 테스트 작동 | Red-Green 사이클 검증 완료 | 테스트가 한 번 통과 |
| Agent 완료 | VCS diff에 변경사항 확인 | Agent가 "성공"이라고 보고 |
| 요구사항 충족 | 항목별 체크리스트 확인 | 테스트 통과 |

## 위험 신호 - 멈춰라

- "~할 것이다", "아마", "~인 것 같다" 사용
- 검증 전에 만족 표현 ("좋아!", "완벽해!", "완료!", 등)
- 검증 없이 커밋/푸시/PR 하려는 순간
- Agent 성공 보고를 그대로 신뢰
- 부분 검증에 의존
- "이번 한 번만"이라고 생각
- 피곤해서 작업을 빨리 끝내고 싶음
- **검증을 실행하지 않고 성공을 암시하는 모든 표현**

## 합리화 방지

| 변명 | 현실 |
|------|------|
| "이제 될 것이다" | 검증을 실행하라 |
| "확신한다" | 확신 ≠ 증거 |
| "이번 한 번만" | 예외 없음 |
| "린터 통과했다" | 린터 ≠ 컴파일러 |
| "Agent가 성공이라고 했다" | 독립적으로 검증하라 |
| "피곤하다" | 피로 ≠ 변명 |
| "부분 검사면 충분하다" | 부분은 아무것도 증명하지 못한다 |
| "다른 표현이니 규칙이 적용되지 않는다" | 문구보다 정신 |

## 핵심 패턴

**테스트:**
```
✅ [테스트 명령 실행] [확인: 34/34 통과] "모든 테스트 통과"
❌ "이제 통과할 것이다" / "올바르게 보인다"
```

**회귀 테스트 (TDD Red-Green):**
```
✅ 작성 → 실행 (통과) → 수정 되돌리기 → 실행 (반드시 실패) → 복원 → 실행 (통과)
❌ "회귀 테스트를 작성했습니다" (red-green 검증 없이)
```

**빌드:**
```
✅ [빌드 실행] [확인: exit 0] "빌드 통과"
❌ "린터 통과했다" (린터는 컴파일을 검사하지 않는다)
```

**요구사항:**
```
✅ 계획 재확인 → 체크리스트 생성 → 각 항목 검증 → 누락 또는 완료 보고
❌ "테스트 통과, 단계 완료"
```

**Agent 위임:**
```
✅ Agent가 성공 보고 → VCS diff 확인 → 변경사항 검증 → 실제 상태 보고
❌ Agent 보고를 신뢰
```

## 이것이 중요한 이유

24건의 실패 기록에서:
- 파트너가 "당신 말을 믿지 않겠다"고 말함 - 신뢰 파괴
- 정의되지 않은 함수가 배포됨 - 크래시 발생
- 누락된 요구사항이 배포됨 - 불완전한 기능
- 거짓 완료 → 방향 전환 → 재작업으로 시간 낭비
- 위반: "정직은 핵심 가치이다. 거짓말하면 교체된다."

## 적용 시점

**항상 다음 전에:**
- 성공/완료 주장의 모든 변형
- 만족의 모든 표현
- 작업 상태에 대한 모든 긍정적 진술
- 커밋, PR 생성, 작업 완료
- 다음 작업으로 이동
- Agent에게 위임

**규칙 적용 대상:**
- 정확한 문구
- 의역과 동의어
- 성공의 암시
- 완료/정확성을 시사하는 모든 커뮤니케이션

## 결론

**검증에 지름길은 없다.**

명령을 실행하라. 출력을 확인하라. 그런 다음에 결과를 주장하라.

이것은 협상 불가다.
