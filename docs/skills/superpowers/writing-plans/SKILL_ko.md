# 계획 작성

## 개요

코드베이스에 대한 사전 지식이 전혀 없고 판단력이 의심스러운 엔지니어를 상정하여 포괄적인 구현 계획을 작성한다. 각 작업에서 어떤 파일을 다뤄야 하는지, 코드, 테스트, 확인해야 할 문서, 테스트 방법 등 필요한 모든 것을 문서화한다. 전체 계획을 한입 크기의 작업으로 나누어 제공한다. DRY. YAGNI. TDD. 자주 커밋한다.

상대가 숙련된 개발자라고 가정하되, 우리의 도구나 문제 도메인에 대해서는 거의 모른다고 가정한다. 좋은 테스트 설계에 대해서도 잘 모른다고 가정한다.

**시작 시 안내:** "writing-plans 스킬을 사용하여 구현 계획을 작성합니다."

**맥락:** 이 작업은 전용 워크트리(brainstorming 스킬로 생성된)에서 실행되어야 한다.

**계획 저장 위치:** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`
- (사용자가 계획 위치를 지정한 경우 해당 설정이 이 기본값을 대체함)

## 범위 확인

스펙이 여러 독립적인 서브시스템을 다루는 경우, brainstorming 단계에서 하위 프로젝트 스펙으로 분리했어야 한다. 분리하지 않았다면, 서브시스템별로 별도의 계획으로 나눌 것을 제안한다. 각 계획은 독립적으로 동작하고 테스트 가능한 소프트웨어를 산출해야 한다.

## 파일 구조

작업을 정의하기 전에, 생성하거나 수정할 파일을 정리하고 각 파일의 역할을 명시한다. 분해 결정이 여기서 확정된다.

- 명확한 경계와 잘 정의된 인터페이스를 가진 단위로 설계한다. 각 파일은 하나의 명확한 책임을 가져야 한다.
- 한 번에 컨텍스트에 담을 수 있는 코드에 대해 가장 잘 추론할 수 있으며, 파일이 집중적일 때 편집이 더 안정적이다. 너무 많은 일을 하는 큰 파일보다 작고 집중된 파일을 선호한다.
- 함께 변경되는 파일은 함께 위치해야 한다. 기술 계층이 아닌 책임 기준으로 분리한다.
- 기존 코드베이스에서는 확립된 패턴을 따른다. 코드베이스가 큰 파일을 사용하는 경우 일방적으로 구조를 변경하지 않는다. 다만 수정 중인 파일이 지나치게 비대해졌다면 분할을 계획에 포함하는 것은 합리적이다.

이 구조가 작업 분해에 영향을 준다. 각 작업은 독립적으로 의미 있는 자체 완결적 변경을 산출해야 한다.

## 한입 크기 작업 단위

**각 단계는 하나의 행동(2-5분):**
- "실패하는 테스트 작성" - 단계
- "실패하는지 확인하기 위해 실행" - 단계
- "테스트를 통과시키는 최소한의 코드 구현" - 단계
- "테스트를 실행하여 통과하는지 확인" - 단계
- "커밋" - 단계

## 계획 문서 헤더

**모든 계획은 반드시 이 헤더로 시작해야 한다:**

```markdown
# [기능 이름] 구현 계획

> **에이전트 작업자 참고:** 필수 하위 스킬: 이 계획을 작업별로 구현하려면 superpowers:subagent-driven-development (권장) 또는 superpowers:executing-plans를 사용하세요. 단계는 체크박스(`- [ ]`) 구문으로 추적합니다.

**목표:** [이것이 무엇을 만드는지 한 문장으로 설명]

**아키텍처:** [접근 방식에 대한 2-3문장]

**기술 스택:** [주요 기술/라이브러리]

---
```

## 작업 구조

````markdown
### Task N: [컴포넌트 이름]

**파일:**
- 생성: `exact/path/to/file.py`
- 수정: `exact/path/to/existing.py:123-145`
- 테스트: `tests/exact/path/to/test.py`

- [ ] **Step 1: 실패하는 테스트 작성**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **Step 2: 테스트를 실행하여 실패하는지 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상 결과: FAIL - "function not defined"

- [ ] **Step 3: 최소한의 구현 작성**

```python
def function(input):
    return expected
```

- [ ] **Step 4: 테스트를 실행하여 통과하는지 확인**

실행: `pytest tests/path/test.py::test_name -v`
예상 결과: PASS

- [ ] **Step 5: 커밋**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 플레이스홀더 금지

모든 단계는 엔지니어에게 필요한 실제 내용을 포함해야 한다. 다음은 **계획의 실패**에 해당하므로 절대 작성하지 않는다:
- "TBD", "TODO", "나중에 구현", "세부 사항 채울 것"
- "적절한 에러 처리 추가" / "유효성 검사 추가" / "엣지 케이스 처리"
- "위의 내용에 대한 테스트 작성" (실제 테스트 코드 없이)
- "Task N과 유사" (코드를 반복 기재하라 - 엔지니어가 작업을 순서대로 읽지 않을 수 있다)
- 방법을 보여주지 않고 무엇을 해야 하는지만 설명하는 단계 (코드 단계에는 코드 블록 필수)
- 어떤 작업에서도 정의되지 않은 타입, 함수 또는 메서드에 대한 참조

## 기억할 점
- 항상 정확한 파일 경로
- 모든 단계에 완전한 코드 - 단계에서 코드를 변경하면 코드를 보여준다
- 예상 출력이 포함된 정확한 명령
- DRY, YAGNI, TDD, 자주 커밋

## 자체 검토

전체 계획 작성 후, 새로운 시각으로 스펙을 다시 보고 계획과 대조 확인한다. 이것은 직접 수행하는 체크리스트이며, 서브에이전트에게 위임하는 것이 아니다.

**1. 스펙 커버리지:** 스펙의 각 섹션/요구사항을 훑어본다. 이를 구현하는 작업을 가리킬 수 있는가? 누락된 부분을 나열한다.

**2. 플레이스홀더 스캔:** 위의 "플레이스홀더 금지" 섹션에 나온 패턴 중 빨간 신호가 있는지 계획을 검색한다. 발견하면 수정한다.

**3. 타입 일관성:** 이후 작업에서 사용한 타입, 메서드 시그니처, 속성 이름이 이전 작업에서 정의한 것과 일치하는가? Task 3에서 `clearLayers()`라고 한 함수를 Task 7에서 `clearFullLayers()`라고 하면 버그다.

문제를 발견하면 인라인으로 수정한다. 재검토는 필요 없으며, 수정 후 진행한다. 작업이 없는 스펙 요구사항을 발견하면 작업을 추가한다.

## 실행 핸드오프

계획 저장 후, 실행 방식을 선택하도록 제안한다:

**"계획이 완료되어 `docs/superpowers/plans/<filename>.md`에 저장되었습니다. 두 가지 실행 옵션이 있습니다:**

**1. 서브에이전트 기반 (권장)** - 작업마다 새로운 서브에이전트를 파견하고, 작업 사이에 리뷰하며, 빠른 반복

**2. 인라인 실행** - 이 세션에서 executing-plans를 사용하여 작업을 실행, 체크포인트가 포함된 일괄 실행

**어떤 방식을 선택하시겠습니까?"**

**서브에이전트 기반 선택 시:**
- **필수 하위 스킬:** superpowers:subagent-driven-development 사용
- 작업마다 새로운 서브에이전트 + 2단계 리뷰

**인라인 실행 선택 시:**
- **필수 하위 스킬:** superpowers:executing-plans 사용
- 리뷰를 위한 체크포인트가 포함된 일괄 실행
