# 체계적 디버깅

## 개요

무작위 수정은 시간을 낭비하고 새로운 버그를 만든다. 임시 패치는 근본 문제를 은폐한다.

**핵심 원칙:** 수정을 시도하기 전에 반드시 근본 원인을 찾아라. 증상 수정은 실패다.

**이 프로세스의 문구를 위반하는 것은 디버깅의 정신을 위반하는 것이다.**

## 철칙

```
근본 원인 조사 없이 수정 금지
```

Phase 1을 완료하지 않았다면 수정을 제안할 수 없다.

## 사용 시점

모든 기술적 이슈에 사용하라:
- 테스트 실패
- 프로덕션 버그
- 예상치 못한 동작
- 성능 문제
- 빌드 실패
- 통합 이슈

**특히 다음 상황에서 사용하라:**
- 시간 압박을 받을 때 (긴급 상황에서는 추측이 유혹적이다)
- "빠른 수정 하나만"이 명백해 보일 때
- 이미 여러 번 수정을 시도했을 때
- 이전 수정이 동작하지 않았을 때
- 이슈를 완전히 이해하지 못했을 때

**다음 경우에도 건너뛰지 마라:**
- 이슈가 단순해 보일 때 (단순한 버그에도 근본 원인이 있다)
- 급할 때 (서두르면 재작업이 보장된다)
- 관리자가 지금 당장 수정하라고 할 때 (체계적 접근이 삽질보다 빠르다)

## 4단계

각 단계를 완료한 후에야 다음 단계로 진행해야 한다.

### Phase 1: 근본 원인 조사

**수정을 시도하기 전에:**

1. **에러 메시지를 주의 깊게 읽어라**
   - 에러나 경고를 건너뛰지 마라
   - 대부분 정확한 해결책을 포함하고 있다
   - 스택 트레이스를 완전히 읽어라
   - 라인 번호, 파일 경로, 에러 코드를 기록하라

2. **일관되게 재현하라**
   - 확실하게 트리거할 수 있는가?
   - 정확한 단계는 무엇인가?
   - 매번 발생하는가?
   - 재현할 수 없다면 → 더 많은 데이터를 수집하고, 추측하지 마라

3. **최근 변경사항을 확인하라**
   - 이 문제를 유발할 수 있는 변경이 무엇인가?
   - Git diff, 최근 커밋
   - 새로운 의존성, 설정 변경
   - 환경 차이

4. **다중 컴포넌트 시스템에서 증거를 수집하라**

   **시스템에 여러 컴포넌트가 있을 때 (CI → 빌드 → 서명, API → 서비스 → 데이터베이스):**

   **수정을 제안하기 전에 진단 계측을 추가하라:**
   ```
   각 컴포넌트 경계에 대해:
     - 컴포넌트에 들어오는 데이터를 로깅
     - 컴포넌트에서 나가는 데이터를 로깅
     - 환경/설정 전파를 검증
     - 각 계층의 상태를 확인

   한 번 실행하여 어디서 깨지는지 보여주는 증거를 수집
   그런 다음 증거를 분석하여 실패하는 컴포넌트를 식별
   그런 다음 해당 특정 컴포넌트를 조사
   ```

   **예시 (다중 계층 시스템):**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **이것이 밝히는 것:** 어느 계층이 실패하는가 (secrets → workflow 성공, workflow → build 실패)

5. **데이터 흐름을 추적하라**

   **에러가 콜 스택 깊은 곳에 있을 때:**

   완전한 역추적 기법은 이 디렉토리의 `root-cause-tracing.md`를 참고하라.

   **간단한 버전:**
   - 잘못된 값은 어디서 발생하는가?
   - 무엇이 잘못된 값으로 이것을 호출했는가?
   - 소스를 찾을 때까지 계속 위로 추적하라
   - 증상이 아닌 소스에서 수정하라

### Phase 2: 패턴 분석

**수정하기 전에 패턴을 찾아라:**

1. **동작하는 예제를 찾아라**
   - 같은 코드베이스에서 유사하게 동작하는 코드를 찾아라
   - 깨진 것과 유사한 것 중 동작하는 것은 무엇인가?

2. **참조와 비교하라**
   - 패턴을 구현하는 경우, 참조 구현을 완전히 읽어라
   - 대충 훑지 마라 - 모든 줄을 읽어라
   - 적용하기 전에 패턴을 완전히 이해하라

3. **차이점을 식별하라**
   - 동작하는 것과 깨진 것의 차이가 무엇인가?
   - 아무리 작더라도 모든 차이점을 나열하라
   - "그건 상관없겠지"라고 가정하지 마라

4. **의존성을 이해하라**
   - 이것이 필요로 하는 다른 컴포넌트는 무엇인가?
   - 어떤 설정, 구성, 환경이 필요한가?
   - 어떤 가정을 하고 있는가?

### Phase 3: 가설과 테스트

**과학적 방법:**

1. **단일 가설을 세워라**
   - 명확하게 서술하라: "Y 때문에 X가 근본 원인이라고 생각한다"
   - 적어두라
   - 모호하지 않고 구체적이어야 한다

2. **최소한으로 테스트하라**
   - 가설을 테스트하기 위해 가능한 가장 작은 변경을 하라
   - 한 번에 하나의 변수만
   - 여러 것을 동시에 수정하지 마라

3. **계속하기 전에 검증하라**
   - 동작했는가? 예 → Phase 4
   - 동작하지 않았는가? 새로운 가설을 세워라
   - 위에 더 많은 수정을 추가하지 마라

4. **모를 때는**
   - "X를 이해하지 못하겠다"고 말하라
   - 아는 척하지 마라
   - 도움을 요청하라
   - 더 조사하라

### Phase 4: 구현

**증상이 아닌 근본 원인을 수정하라:**

1. **실패하는 테스트 케이스를 만들어라**
   - 가능한 가장 단순한 재현
   - 가능하면 자동화된 테스트
   - 프레임워크가 없다면 일회용 테스트 스크립트
   - 수정 전에 반드시 있어야 한다
   - 적절한 실패 테스트 작성은 `superpowers:test-driven-development` 스킬을 사용하라

2. **단일 수정을 구현하라**
   - 식별된 근본 원인을 해결하라
   - 한 번에 하나의 변경만
   - "하는 김에" 개선하지 마라
   - 리팩토링을 끼워넣지 마라

3. **수정을 검증하라**
   - 테스트가 통과하는가?
   - 다른 테스트가 깨지지 않았는가?
   - 이슈가 실제로 해결되었는가?

4. **수정이 동작하지 않을 때**
   - 멈춰라
   - 세어봐라: 몇 번의 수정을 시도했는가?
   - 3번 미만이면: Phase 1로 돌아가서 새로운 정보로 재분석하라
   - **3번 이상이면: 멈추고 아키텍처를 의심하라 (아래 5단계)**
   - 아키텍처 논의 없이 4번째 수정을 시도하지 마라

5. **3번 이상 수정에 실패했을 때: 아키텍처를 의심하라**

   **아키텍처 문제를 나타내는 패턴:**
   - 각 수정이 다른 곳에서 새로운 공유 상태/결합/문제를 드러낸다
   - 수정을 구현하려면 "대규모 리팩토링"이 필요하다
   - 각 수정이 다른 곳에서 새로운 증상을 만든다

   **멈추고 기본 전제를 의심하라:**
   - 이 패턴이 근본적으로 건전한가?
   - "관성으로 그냥 유지하고 있는" 것은 아닌가?
   - 증상 수정을 계속할 것인가, 아키텍처를 리팩토링할 것인가?

   **더 많은 수정을 시도하기 전에 파트너와 논의하라**

   이것은 실패한 가설이 아니다 - 잘못된 아키텍처다.

## 위험 신호 - 멈추고 프로세스를 따르라

다음과 같은 생각을 하고 있다면:
- "일단 급한 대로 수정하고, 나중에 조사하자"
- "X를 바꿔보고 되는지 보자"
- "여러 변경을 추가하고 테스트를 돌리자"
- "테스트는 건너뛰고 수동으로 확인하자"
- "아마 X일 거야, 그걸 수정하자"
- "완전히 이해하지 못하지만 이게 될 수도 있어"
- "패턴은 X라고 하지만 나는 다르게 적용하겠어"
- "주요 문제들은 이거다: [조사 없이 수정 목록을 나열]"
- 데이터 흐름을 추적하기 전에 해결책을 제안
- **"한 번만 더 수정 시도" (이미 2회 이상 시도한 경우)**
- **각 수정이 다른 곳에서 새로운 문제를 드러냄**

**이 모든 것은 의미한다: 멈춰라. Phase 1로 돌아가라.**

**3번 이상 수정에 실패했다면:** 아키텍처를 의심하라 (Phase 4.5 참고)

## 파트너가 잘못하고 있다는 신호

**다음과 같은 지적을 주시하라:**
- "그게 안 되고 있는 거 아냐?" - 검증 없이 가정했다
- "그걸로 알 수 있을까...?" - 증거 수집을 추가했어야 한다
- "추측 그만해" - 이해 없이 수정을 제안하고 있다
- "이거 깊이 생각해봐" - 증상이 아니라 기본 전제를 의심하라
- "우리 막힌 거야?" (좌절) - 당신의 접근 방식이 동작하지 않고 있다

**이런 것을 볼 때:** 멈춰라. Phase 1로 돌아가라.

## 흔한 합리화

| 변명 | 현실 |
|------|------|
| "이슈가 단순해서 프로세스가 필요 없어" | 단순한 이슈에도 근본 원인이 있다. 단순한 버그에는 프로세스가 빠르다. |
| "긴급이라 프로세스 할 시간이 없어" | 체계적 디버깅이 추측-확인 삽질보다 빠르다. |
| "일단 이것부터 해보고, 그 다음에 조사하자" | 첫 번째 수정이 패턴을 만든다. 처음부터 제대로 하라. |
| "수정이 되는지 확인 후에 테스트를 작성하겠다" | 테스트되지 않은 수정은 지속되지 않는다. 먼저 테스트해야 증명된다. |
| "여러 수정을 한 번에 하면 시간이 절약된다" | 무엇이 동작했는지 분리할 수 없다. 새로운 버그를 유발한다. |
| "참조가 너무 길어서 패턴을 적용하겠다" | 불완전한 이해는 버그를 보장한다. 완전히 읽어라. |
| "문제가 보인다, 수정하겠다" | 증상을 보는 것은 근본 원인을 이해하는 것과 다르다. |
| "한 번만 더 수정 시도" (2번 이상 실패 후) | 3번 이상 실패 = 아키텍처 문제. 패턴을 의심하라, 다시 수정하지 마라. |

## 빠른 참조

| 단계 | 주요 활동 | 성공 기준 |
|------|----------|----------|
| **1. 근본 원인** | 에러 읽기, 재현, 변경사항 확인, 증거 수집 | 무엇이 왜 그런지 이해 |
| **2. 패턴** | 동작하는 예제 찾기, 비교 | 차이점 식별 |
| **3. 가설** | 이론 수립, 최소한으로 테스트 | 확인되었거나 새로운 가설 |
| **4. 구현** | 테스트 작성, 수정, 검증 | 버그 해결, 테스트 통과 |

## 프로세스가 "근본 원인 없음"을 밝힐 때

체계적 조사 결과 이슈가 진정으로 환경적이거나, 타이밍 의존적이거나, 외부 요인일 때:

1. 프로세스를 완료한 것이다
2. 조사한 내용을 문서화하라
3. 적절한 처리를 구현하라 (재시도, 타임아웃, 에러 메시지)
4. 향후 조사를 위한 모니터링/로깅을 추가하라

**그러나:** "근본 원인 없음"의 95%는 불완전한 조사다.

## 보조 기법

이 기법들은 체계적 디버깅의 일부이며 이 디렉토리에서 사용할 수 있다:

- **`root-cause-tracing.md`** - 콜 스택을 역추적하여 원래 트리거를 찾는 버그 추적
- **`defense-in-depth.md`** - 근본 원인을 찾은 후 여러 계층에 유효성 검사 추가
- **`condition-based-waiting.md`** - 임의의 타임아웃을 조건 폴링으로 교체

**관련 스킬:**
- **superpowers:test-driven-development** - 실패하는 테스트 케이스 생성 (Phase 4, 1단계)
- **superpowers:verification-before-completion** - 성공을 선언하기 전에 수정이 동작했는지 검증

## 실제 영향

디버깅 세션에서:
- 체계적 접근: 수정까지 15-30분
- 무작위 수정 접근: 2-3시간 삽질
- 첫 번째 시도 수정 성공률: 95% vs 40%
- 도입된 새로운 버그: 거의 없음 vs 흔함
